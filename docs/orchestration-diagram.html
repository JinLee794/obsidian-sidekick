1 Key Files & What They Do
src/configLoader.ts
Config Loader
Reads *.agent.md, */SKILL.md, mcp.json, *.prompt.md, *.trigger.md, *.instructions.md from vault folders. Parses YAML frontmatter + markdown body. Returns typed arrays.
src/view/sessionConfig.ts
Session Config Builder
Assembles the SessionConfig object passed to the SDK: model, MCP servers, skills, agents, system message, tool exclusions, BYOK provider.
src/view/chatSession.ts
Chat Session Manager
Owns the full send flow: prompt building → triage → session creation → session.send(). Registers all SDK event listeners. Manages session lifecycle.
src/copilot.ts
SDK Wrapper
Thin wrapper around @github/copilot-sdk. Exposes createSession(), chat(), listSessions(), resumeSession(), listTools().
src/view/chatRenderer.ts
Chat Renderer
Renders streamed content, tool call blocks, subagent blocks, token usage, skill badges, and turn metadata footers in the chat DOM.
src/sidekickView.ts
Sidekick View (Main)
The Obsidian ItemView. Orchestrates everything: stores loaded configs, manages state, creates the sidebar UI, dispatches to mixins.
src/types.ts
Type Definitions
All shared interfaces: AgentConfig, SkillInfo, McpServerEntry, TriggerConfig, PromptConfig, etc.
src/settings.ts
Settings & Scaffold
Plugin settings tab. Controls folder paths (sidekick/agents, sidekick/skills, etc.), context mode, triage toggle, tool approval policy.

4 System Message Assembly (what the LLM actually sees)

Built in sessionConfig.ts → buildSessionConfig(). Each layer is appended with \n\n separator. The final string is sent as systemMessage: {mode: 'append', content} — meaning it's appended to whatever the SDK's default system prompt is, not replacing it.

1 Global Instructions
Concatenated content from all *.instructions.md files in the vault config folder.
Source: this.globalInstructions (loaded by configLoader.ts → loadInstructions())
2 MCP Preference Rule
"Prefer MCP tools over bash/shell for external API calls. If a tool call fails, report the error — do not retry."
Condition: only if mcpServers.length > 0
3 MCP Tool Catalog
Full list of discovered MCP tools with names + descriptions, grouped by server.
Example: MCP server "m365" tools:\n - mail:ListMessages: List emails...
Source: this.mcpServerStatus Map (populated after session.info events)
4 Subagent Failure Guidance
"If a subagent fails, report the failure immediately — do not retry or spawn additional subagents."
Condition: only if customAgents.length > 0
5 ⭐ Skill Catalog + Priority Hint
Lists all enabled skills with names + descriptions.
Includes: "IMPORTANT: When a request matches an available skill, invoke that skill instead of performing ad-hoc tool operations (grep, view, bash, etc.)."
Source: this.skills filtered by this.enabledSkills — recently added to fix skill triggering
6 Context Mode Guidance
"Use on-demand file/search tools to gather vault context when needed."
Condition: only if contextMode === 'suggest'
📌 Note: Agent-specific instructions are NOT in the system message. They're passed separately in customAgents[].prompt — the SDK handles injecting the selected agent's instructions during its own orchestration.
5 SDK Events & What Handles Them
Event Type Handler Location What Happens
assistant.message_delta Stream chatSession.ts Appends text chunk → re-renders markdown in streaming body
tool.execution_start Tool chatSession.ts → chatRenderer.ts Creates collapsible tool call block (name, args, spinner)
tool.execution_complete Tool chatSession.ts → chatRenderer.ts Removes spinner, shows result/error in tool block
skill.invoked Skill chatSession.ts Records skill name in turnSkillsUsed[]. Debug-logged.
subagent.started Agent chatSession.ts → chatRenderer.ts Creates subagent block with name badge + activity counter
subagent.completed Agent chatSession.ts → chatRenderer.ts Marks subagent block as done (green check)
subagent.failed Agent chatSession.ts → chatRenderer.ts Marks subagent block as failed (red X + error)
subagent.selected Agent chatSession.ts Debug-logged only — SDK internally routes to selected agent
assistant.usage Meta chatSession.ts Accumulates input/output/cache tokens for footer display
session.idle Meta chatSession.ts → chatRenderer.ts Finalizes streaming: renders footer (tokens, tools, skills), shows handoff buttons
session.info / .warning Meta chatSession.ts MCP server status updates (connection, tool discovery)

8 Agent Routing, Triage & Handoffs
User sends message
No agent pre-selected
Triage LLM Call
sessionConfig.ts
"Given these agents:
- researcher: Find papers
- writer: Draft content
Which fits this request?"
→ Returns agent name or "none"
Session Rebuilt
triageAgentForSession = 'writer'
configDirty = true
→ ensureSession() creates new session
with agent-specific config
Handoffs work differently from triage. When an agent defines handoffs: [{agent: 'writer', label: 'Draft it'}], buttons appear after the response. Clicking one switches selectedAgent, sets configDirty = true, and optionally auto-sends a prompt. The session is rebuilt with the new agent's config.
SDK-level delegation: All agents are always in customAgents[] regardless of selection. The SDK can internally delegate to subagents (triggering subagent.started / subagent.completed events). This is separate from triage/handoff — it's the LLM deciding to use another agent as a "tool".
9 Background & Trigger Sessions
Session Switching (sidebar)
sessionSidebar.ts
When you switch sessions:
1. Current session's DOM saved to BackgroundSession.savedDom
2. Events stay subscribed — streaming continues in background
3. Target session's DOM restored or new session created
4. turnSkillsUsed, turnToolsUsed, token accumulators preserved
Trigger-Based Sessions
triggersPanel.ts + triggerScheduler.ts
Cron triggers: Polled every 60s, fires when schedule matches.
Glob triggers: File change watcher, 5s debounce per trigger.
Both create a background session with the trigger's agent + prompt.
Uses same buildSessionConfig() pipeline — skills included.
10 Debug Checklist — "Why isn't my skill triggering?"
  1. Enable debug mode in plugin settings → check console for debugTrace('buildSessionConfig', ...)
  2. Verify enabledSkills contains your skill name (logged in trace)
  3. Verify skillDirectories is not empty (logged in trace)
  4. Verify skipSkills is false (it's true when using /prompt templates)
  5. Check the system message in the SDK request — does it list your skill in "Available skills"?
  6. If using an agent, check the agent's skills: frontmatter — empty [] disables all skills
  7. Watch for debugTrace('skill.invoked', {name}) in console — if absent, the SDK chose not to invoke it
  8. Check if the skill's SKILL.md has valid YAML frontmatter with name and description
  9. Ensure the skill folder is inside the configured skills/ path (check Settings tab)
